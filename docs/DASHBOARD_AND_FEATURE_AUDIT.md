# NoAgent4U Dashboard and Feature Audit

Audit date: 2026-07-19
Repository state reviewed: `agent/fix-production-migrations` at `bf85be8`
Production URL verified read-only: `https://noagent4u.com`

## Executive summary

The deployed application is a genuine, database-backed foundation rather than a mock. Authentication, organization records, property creation/listing, lease/billing/invoice/payment domain services, and an outbox worker all exist. However, none of the requested administration, owner, or tenant experiences is complete end to end under the definition in the brief. The authenticated web app is one shared organization shell with six routes; it is not three role-specific dashboards.

The most urgent issue is authorization design. Organization membership is checked, but property-owner scope is not. A user with one of three hard-coded role codes (`OWNER`, `ADMIN`, `PROPERTY_MANAGER`) can reach organization-wide property, lease, invoice, and payment data. `PropertyOwnership` exists but is not used to constrain those reads. The seeded permission model and service-level role checks also conflict: the finance role cannot use finance operations, while a property manager receives broad finance access. Tenant and lease-party records are not linked strongly enough to an authenticated user to implement secure tenant self-service.

Production verification found real data on Dashboard and Properties. Leases, Invoices, and Payments returned real empty collections. Settings failed with `Organization settings not found`. The tested session was the only available role context, so role-by-role live verification is classified `NOT VERIFIED`; code-level authorization findings remain actionable.

Overall assessment: a useful backend foundation with an early management UI, but substantial P0 security/data-model work and P1 workflow development are required before the platform should be described as a complete rental-management product.

| `FINANCE`              | Seed grants only identity profile permission; management services do not accept this role code  | Cannot perform intended finance work                     | Role name and actual authorization conflict. `BROKEN`.                                                                                                              |
| `TENANT`               | Membership/profile permissions only                                                             | No tenant dashboard or self-service domain linkage       | Lease parties are contact snapshots, not authenticated principals. `MISSING` as a usable role experience.                                                           |
| Property owner record  | `PropertyOwnership` model records ownership percentage                                          | Created at property creation                             | Not applied to query authorization. `BACKEND ONLY`.                                                                                                                 |

Two authorization mechanisms currently coexist:

1. `PermissionGuard` and `@RequirePermissions`, used principally by organization settings/role flows.
2. Service-level checks of system role codes in property, lease, billing, invoice, and payment services.

They must be consolidated into explicit API policies that combine organization membership, resource scope, action, and role/permission. Hiding menus is not authorization.

## Broken and incomplete workflows

1. **Settings bootstrap:** the production settings record is absent. A settings endpoint should either create defaults transactionally for an authorized organization or a safe idempotent migration should backfill every organization. Returning a hard 404 makes the main route unusable.
2. **Property-to-operations workflow:** a property can be created, but the web UI cannot open it, add buildings/units, assign an owner, create leases, or see finances.
3. **Lease-to-cash workflow:** backend pieces exist for lease, billing schedule, invoice, and payment, but no operator UI connects the sequence and no scheduler automates it.
4. **Tenant identity:** a lease party and unit occupant are unverified contact snapshots. They cannot safely authorize tenant access or guarantee that invoices/payments shown belong to the signed-in person.
5. **Document workflow:** APIs register storage keys but do not provide upload, signed download, malware/content controls, or template-generated documents.
6. **Notification workflow:** events can be persisted, but the running worker does not compose usable SMTP/SMS/WhatsApp delivery. Organization handlers explicitly terminate as no-ops.
7. **Finance access:** a seeded finance user is unable to use finance services; a property manager can use them too broadly.

## Security and data-isolation findings

### P0

- **Missing ownership scope:** `PropertyOwnership` is not an authorization boundary. In a management organization with multiple owners, organization-wide queries can disclose other owners' properties and related financial/tenant data.
- **Missing tenant scope:** there is no trustworthy user-to-lease-party relation, so tenant self-service must not be implemented by email-string matching.
- **Inconsistent RBAC:** role-code allowlists in domain services bypass the permission model. This creates both privilege expansion and denial of intended finance access.
- **Platform/tenant principal coupling:** the platform super administrator is represented through organization membership. Complete `ADR-O1-001` before expanding the platform console.
- **Settings production failure:** not a direct disclosure, but a critical broken configuration path. Backfill must be idempotent and production-safe.

### P1/P2 hardening

- Add API-level negative tests for cross-organization, cross-property-owner, and cross-tenant access on list and detail routes.
- Create explicit resource policies (`organization`, `property`, `lease`, `invoice`, `payment`, `document`) and deny by default.
- Do not return stored integration secrets. Encrypt credentials with key rotation metadata and audit all config changes.
- Verify gateway webhook signatures before parsing into business events; persist provider event IDs under a unique constraint.
- Use signed, short-lived storage URLs and validate file type/size; never authorize solely from a client-provided storage key.
- Consider PostgreSQL row-level security only as later defense in depth after application policies and connection strategy are proven.

## Performance observations

- Dashboard issues multiple list calls to calculate four totals. A scoped aggregate endpoint/read model will reduce transfer and repeated query cost and provide a consistent time boundary.
- Required property/invoice/collection metrics span several aggregates; do not assemble large unbounded datasets in the browser.
- Existing API pagination is useful, but Properties does not expose it and generic lists provide only rudimentary previous/next controls.
- Add indexes only after query plans are defined for owner scope, tenant scope, lease expiry, invoice aging, and reconciliation queues.

## Decisions requiring product-owner approval

1. Confirm that platform administrators and organization administrators are distinct principals and workspaces.
2. Define whether `OWNER` means organization proprietor, managed-property asset owner, or both. The recommended model separates these concepts.
3. Approve a property assignment model for property managers and finance staff (all properties versus explicit portfolio).
4. Approve tenant identity onboarding: invitation to an existing `Person/User` with a verified lease-party link.
5. Approve canonical lease/payment statuses and tax/GST requirements before migrations.
6. Select the first email/SMS/WhatsApp and payment providers only after adapter/security requirements are accepted.
7. Decide retention rules for documents, delivery logs, audit events, and personally identifiable tenant data.

## Conclusion

The current pages use real APIs and real production data; they are not mock dashboards. They are, however, early operational surfaces. Separate administration, property-owner, and tenant dashboards do not currently exist. Implementation should begin only after the P0 principal, role, ownership, and tenant-linkage decisions are approved.
