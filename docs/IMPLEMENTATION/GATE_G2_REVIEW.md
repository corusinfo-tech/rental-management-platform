# Gate G2 — Identity Platform Architecture & Security Review

## Decision

**FAIL.** The platform has a coherent Identity direction and passing local checks, but it is not safe or complete enough to declare the Identity Platform production-ready. Critical authentication, OTP, delivery, and test-infrastructure gaps must close before the next gate.

## Scope and evidence

Reviewed the active Identity module, Prisma schemas/migrations, Docker configuration, implementation reports, ADR-G2.2-001, and local verification outputs. `AGENTS.md` and a checkout-local `docs/ENGINEERING_HANDBOOK` were not present; the approved Handbook rules previously supplied through Google Docs and the implementation reports were used as the available authority.

Local evidence: Prisma format/validate/generate, lint, typecheck, build, and API unit tests passed most recently. The database suite remains skipped without `IDENTITY_TEST_DATABASE_URL`; Docker was unavailable. No clean migration, PostgreSQL integration, Redis integration, or delivery-worker test was executed.

## Critical findings

### G2-C01 — OTPs use a non-cryptographic random source

**Evidence:** `apps/api/src/identity/verification-engine/verification-channel.ts` generates SMS/WhatsApp OTPs with `Math.random()`.

**Impact:** A six-digit OTP generated by a predictable, non-CSPRNG source is not suitable for authentication/verification. This violates the security expectation for verification secrets and can enable OTP prediction.

**Required correction:** Generate OTPs with `node:crypto` (`randomInt` or rejection-sampling over cryptographically random bytes); test uniqueness/range and retain the existing attempt-limit controls.

### G2-C02 — Verification cannot be delivered end-to-end

**Evidence:** `VerificationDeliveryEnvelope` and ID-only outbox events exist, but no outbox publisher, BullMQ worker, claiming/completion lifecycle, or concrete authorised provider is implemented. All relevant reports explicitly defer worker/provider delivery.

**Impact:** Registration, password reset, SMS, and WhatsApp verification tokens are never delivered. The user-visible identity flows cannot complete in a deployed system.

**Required correction:** Implement the approved worker/outbox handoff, idempotent claiming/completion, retry/DLQ, envelope decryption boundary, provider abstraction integration, and post-delivery destruction before production.

### G2-C03 — Session revocation does not invalidate access tokens

**Evidence:** `apps/api/src/identity/authorization/access-token.guard.ts` validates JWT signature/issuer/audience only. It does not load or check the session ID (`sid`) against revocation/expiry. Logout and password reset only update `Session.revokedAt`.

**Impact:** A logged-out, revoked, or password-reset session’s already-issued access JWT remains accepted until its 15-minute expiry.

**Required correction:** Define and implement the approved access-token revocation model (session lookup/cache deny-list with Redis and fail-mode policy, or explicitly accept bounded access-token lifetime) and test logout/password-reset behavior.

## High findings

### G2-H01 — Login and password-reset request endpoints have no rate limiting

**Evidence:** Only registration and email-verification request throttle services exist. `IdentityController` routes login, password reset, SMS, and WhatsApp requests directly to services.

**Impact:** Password spraying, credential stuffing, account enumeration through timing, OTP flooding, and reset abuse are not controlled according to the Handbook’s authentication API rate-limit requirement.

**Required correction:** Apply privacy-preserving Redis limits by IP and HMAC identifier fingerprint to all authentication/verification request endpoints; document Redis failure mode and test TTL/cooldown behavior.

### G2-H02 — Refresh rotation is not exposed and has incomplete failure auditing

**Evidence:** `IdentityService.refresh()` exists but no `POST /auth/refresh` controller route is present. Invalid JWT/hash/race paths return generic failure but do not consistently create `RefreshFailed` audit records.

**Impact:** The implemented rotation/reuse mechanism is not reachable by the public API and cannot meet the stated refresh lifecycle/audit acceptance criteria.

**Required correction:** Add the approved refresh endpoint/DTO, audit all failure classes without secrets, and test concurrent refresh/reuse against PostgreSQL.

### G2-H03 — Envelope key rotation is fail-closed but not operationally supported

**Evidence:** `VerificationEnvelopeService` accepts only one configured key/version and rejects records with another version. The G2.2 report acknowledges historical key material is not configured.

**Impact:** Rotating `VERIFICATION_KEY_VERSION` makes outstanding envelopes undecryptable; no documented overlap/retention process exists.

**Required correction:** Implement the ADR-approved key-ring/secret-manager retrieval and bounded prior-version support, or define a rotation runbook that intentionally expires/destroys all old envelopes before rollout.

### G2-H04 — Generic engine expiry/cleanup is incomplete

**Evidence:** `VerificationEngine.expire()` updates verification status only; it does not destroy the corresponding envelope, publish `VerificationExpired`, or audit expiry. `cleanup()` only expires envelopes for a user and does not expire verification rows or emit events.

**Impact:** Public engine methods do not uphold the engine’s own lifecycle contract, leaving encrypted material/events/audit inconsistent outside the resend/verify paths.

**Required correction:** Make every engine lifecycle operation transactionally update verification and envelope state, publish the required event, and create the required audit record.

### G2-H05 — Required database and concurrency coverage is absent

**Evidence:** `tests/identity-database.test.mjs` is skipped; reports consistently state Docker/PostgreSQL unavailable. Unit tests use service mocks.

**Impact:** Unique constraints, partial indexes, migration ordering, transaction rollback, concurrent verification/refresh, and session-family revocation are unproven on PostgreSQL.

**Required correction:** Run clean PostgreSQL 16 migration/seed/rollback and integration suites using disposable containers; add Redis tests for rate limits and configured failure handling.

## Medium findings

- **G2-M01 — Transactional audit consistency varies.** Login session creation and login-success audit occur in separate transactions; a partial failure can leave a session without its audit record or return an error after creating a session. Move the session/audit pair into one transaction.
- **G2-M02 — Purpose/channel configuration is not truly per-purpose.** `Environment.verification.email` supplies expiry/cooldown/attempt/resend settings to every engine channel, including SMS and WhatsApp. This does not meet configurable per-purpose behavior.
- **G2-M03 — Repeated opaque-token parsing.** Email, password reset, SMS, and WhatsApp adapters each parse tokens. This is duplicated security-sensitive syntax validation that belongs in the engine/token codec.
- **G2-M04 — Authorization/RBAC is only skeletal.** Guards and membership resolvers exist, but no verified route-level policy matrix or organization-scoped Identity authorization tests prove tenant isolation.
- **G2-M05 — Outbox lacks publisher lifecycle fields/claims.** It has status/attempts but no lock/lease, worker identity, error redaction, retry schedule, or idempotency key proven by implementation.
- **G2-M06 — Audit taxonomy is inconsistent.** Required names such as `LoginSucceeded` are implemented as string variants such as `identity.login.succeeded`; there is no central typed event taxonomy or retention/redaction enforcement.
- **G2-M07 — Session device metadata is incomplete.** `deviceId`, user agent, and IP are stored, but `lastUsedAt` is only set on create and is not updated on authenticated access/refresh in a measured lifecycle.
- **G2-M08 — Mobile state semantics are undefined.** SMS/WhatsApp confirmation emits events but does not record verified-mobile state or a mobile verification timestamp; “update account state where appropriate” is unresolved.

## Low findings

- **G2-L01 — Session DTOs return raw operational metadata without an explicit response allow-list mapper.** A dedicated session response mapper would make future fields safer.
- **G2-L02 — Swagger coverage is uneven.** Endpoint summaries exist, but protected response/error variants and the authentication scheme are not consistently documented.
- **G2-L03 — Migrations are additive and use restrictive foreign keys, which is positive; however, no migration rollback/runbook is present for the latest session changes.**
- **G2-L04 — Organization selection uses the first active membership.** This is deterministic only by creation time and needs an explicit membership/organization-selection contract before multi-organization login is relied upon.

## Positive components

- Prisma uses restrictive foreign keys for Identity relationships and additive migrations.
- Passwords, refresh tokens, and verification secrets use Argon2 hashes rather than plaintext storage.
- Envelope ciphertext is separated from ordinary outbox payloads and binds AAD to verification/user/organization/correlation identifiers.
- JWT verification constrains algorithm, issuer, audience, and secret through ConfigModule validation.
- Controllers primarily use DTOs and delegate business workflows to services.
- Registration and verification engine creation use transactions and ID-only outbox payloads.
- Logout/session ownership filters prevent direct cross-user session revocation in the repository query.

## Recommendations and implementation order

1. Replace OTP `Math.random`, add authentication-wide Redis rate limiting, and decide access-token revocation semantics.
2. Implement and test public refresh flow with PostgreSQL-backed reuse/concurrent-rotation protection.
3. Complete the approved outbox/worker/envelope delivery lifecycle and key rotation contract.
4. Repair engine expiry/cleanup to make all lifecycle operations transactional and auditable.
5. Establish typed audit/outbox event contracts, redaction, retention, leases, and idempotency.
6. Add disposable PostgreSQL/Redis CI integration tests and migration rollback validation.
7. Complete RBAC/organization-selection policy tests before organization-scoped modules proceed.

## Technical debt and future improvements

- Split large single-line controller/service methods into testable policy units after security blockers are fixed.
- Introduce a shared opaque-token codec and typed event constants.
- Add response mappers for sessions and a central audit event schema.
- Define mobile verification state and multi-organization login/membership selection behavior.
- Add observability metrics for verification, delivery envelope, refresh reuse, and outbox lag/failures.
